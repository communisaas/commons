# Profile Basemap Hypergraph

**Status:** Cycles 1–6 closed. Engineering done; behind `FEATURES.PROFILE_BASEMAP=false`. Provider pivoted post-hoc from Stadia ($20/mo) to Carto Positron raster tile stitching (free, attribution baked into the rendered PNG). Antimeridian polygon-tear (F-15/F-16) shipped and verified.
**Date:** 2026-05-11
**Scope:** `src/lib/components/profile/GroundSpatialProof.svelte` and adjacent endpoints. Optional alignment of `src/lib/components/geographic/DistrictMap.svelte`.

## Thesis

`GroundSpatialProof` currently renders an SVG polygon on white. The figure floats — the viewer has no geographic anchor and no sense of where their district sits. The principled answer is to add a basemap layer underneath the existing district + privacy-hex paths. The constraint that has kept that off the page is: tile-server calls leak viewport coordinates to a third party, and the existing tile-less rendering claim ("No third-party map tiles are loaded") sits at `GroundSpatialProof.svelte:165` and is quoted approvingly in `docs/specs/POSTAL-BUBBLE-SPEC.md:1271`.

The pragmatic-cypherpunk reframe: third-party calls are acceptable if (a) the user's browser never originates them and (b) the third party's view is bounded to public, deterministic information (district code + atlas version). A server-rendered raster, fetched once per (district, version) and cached at our own edge, satisfies both. The third party sees `district CA-11, atlas v20260418` and our egress IP — nothing per-user, nothing per-view, nothing about the H3 cell that actually identifies the constituent's neighborhood.

The structural finding: **two providers explicitly cover our model, only one is affordable**. Stadia's `static_cacheable` endpoint exists precisely for "store the image, serve it from your own infra" — that's the cypherpunk-pragmatic pattern named in their docs, not improvised by us.

## Decisions

Each `D*` requires explicit user assent before the cycle that depends on it can start.

- **D1 — Provider.** Stadia Maps Starter ($20/mo), `static_cacheable` endpoint. Honors our cache+redistribute pattern under paid subscription. Lower engineering ceiling than self-host; rollback to tile-less is one feature-flag flip. *Pending.*
- **D2 — Style.** `alidade_smooth` — minimalist greys, low chroma. Matches the document register of the design system; the emerald district stroke and teal hex remain the visual signal on top. *Pending.*
- **D3 — Image dimensions.** 360×220 base passed to Stadia, `@2x` returns 720×440 actual pixels. SVG viewBox stays 360×220; the @2x asset renders sharp at retina. Stadia warns browsers struggle past 1024 px in either dimension — 720×440 stays well under. *Confirmed 2026-05-11.*
- **D4 — Bbox padding.** 6% of the district's longer axis on each side. Wide enough to not crop the district stroke at the edge; tight enough that the privacy hex stays a meaningful dot inside. *Pending.*
- **D5 — Pre-warm vs lazy.** Pre-warm at `publish:source` time. The publish script already iterates every district to produce the GeoJSON files; a per-district basemap fetch joins the same loop. Lazy-fill means the first profile view per district per version pays a Stadia round trip; pre-warm makes the user experience uniform. *Pending.*
- **D6 — Endpoint host.** Same origin as the GeoJSON: `atlas.commons.email/source/{version}/us/cd/cd-{geoid}-base.webp`. This is published-asset territory — the Cloudflare R2 bucket the atlas already uses. Not a SvelteKit `/api/*` route. Mirrors the static-bundle pattern shipped 2026-05-03. *Pending.*
- **D7 — Format.** PNG. Stadia returns PNG natively; transcoding to WebP would require adding `sharp` to the shadow-atlas package's dependency graph for a ~20–30% byte saving on line-art rasters. Not worth the native-binary dep. *Confirmed 2026-05-11.*

## Hypergraph

Nodes are inputs (I), artifacts (A), and decisions (D). Hyperedges (H) bind sets of nodes that share a single review gate — the gate checks coherence across the whole set, not pairwise.

### Inputs

- **I1 — District GEOID.** Already canonical via `displayDistrictToGEOID` (`src/lib/core/shadow-atlas/district-format.ts`).
- **I2 — Atlas version.** Already canonical via `manifest.currentVersion` (`src/lib/core/shadow-atlas/district-bundle.ts:80`).
- **I3 — District bbox.** Derivable at publish time from the GeoJSON in the same loop that writes the bundle.
- **I4 — Stadia API key.** New secret. Lives in publish-time environment only (never CF Pages, never client).

### Artifacts

- **A1 — Publish script extension.** Wherever the per-district GeoJSON is produced (the script that backs the atlas Worker's `/source/{v}/us/cd/cd-{geoid}.geojson` files). Adds a per-district Stadia fetch, sharp transcode to WebP, R2 PUT.
- **A2 — R2 asset.** `cd-{geoid}-base.webp` next to `cd-{geoid}.geojson` for every published version. Immutable, edge-cached by the atlas Worker.
- **A3 — `getDistrictBasemapUrl(displayCode)` helper.** Mirrors `getDistrictBoundary`. Returns the WebP URL for the current atlas version, or `null` if manifest unavailable.
- **A4 — `GroundSpatialProof` `<image>` layer.** Mounted inside the existing `<svg>`, beneath the district + cell paths, at `(0, 0, WIDTH, HEIGHT)`. Visibility tied to a successful preload — the existing tile-less rendering remains the fallback if the image fails.
- **A5 — Projection alignment.** The Stadia request must use the same bbox the SVG fits to. Currently `boundsFor(districtRings)` computes the SVG fit; that same bbox (in lat/lng) becomes the Stadia `bbox` parameter. The seam is exact identity, not approximate.
- **A6 — Feature flag.** `FEATURES.PROFILE_BASEMAP` (default `false` until D1 is paid). When false, render the existing tile-less SVG. When true, render with `<image>` underneath.
- **A7 — Publish-time quota guardrail.** A counter in the publish script that aborts the run if cumulative Stadia requests in the current calendar month exceed the configured monthly budget (default 450, leaves headroom under the 500 cap).

### Hyperedges

- **H1 = {I1, I2, I3, A1, A2, D6}.** *Publish surface.* The publish script reads I1/I2/I3, calls Stadia, writes A2 to the URL pattern in D6. Review gate: the URL pattern matches the existing atlas asset convention; a missing per-district render must degrade to A4's fallback path, not 404 the profile page.
- **H2 = {A3, A4, A5}.** *Render surface.* Helper resolves URL; image mounts; bbox-fit identity holds so the basemap and the SVG paths register pixel-exactly. Review gate: a deliberate 1-pixel offset test confirms the image and the district polygon trace the same line — projection drift means the basemap is unsafe to ship.
- **H3 = {D1, D5, I4, A1, A7}.** *Cost surface.* Provider choice, quota model, key location, and abort guardrail must agree. Review gate: a publish dry-run reports exact request count; if it exceeds budget, the run aborts before any third-party call.
- **H4 = {A4, A6, existing tile-less branch}.** *Rollback surface.* The feature flag must cleanly fall back to the existing rendering on any failure (missing asset, image load error, Stadia outage during publish). Review gate: forced flag=false renders identically to the pre-change state.
- **H5 = {existing `DistrictMap.svelte`, the principle stated in `POSTAL-BUBBLE-SPEC.md:1271`}.** *Codebase alignment.* `DistrictMap.svelte:232` currently pulls Carto tiles directly client-side, quietly contradicting the no-client-tile-request claim. If H1–H4 ship, this seam closes with one more endpoint, or the claim is honestly downgraded. Out of scope for the first cycle; named here so the contradiction does not silently widen.

### Cycle 4 — Basemap completeness gate ✓ closed

**Shipped (voter-protocol).** Inline gate in `scripts/publish-source.ts` between basemap rendering and bundle-list population. For each `{country}/{layer}/` directory, the gate reads `index.json` and checks the layer's basemap state:

- No `-base.png` files anywhere in the layer → gate is silent (intent: no basemaps, publish proceeds).
- At least one `-base.png` → every district in the index must have one; otherwise the publish exits 2 before the manifest pointer flips.

The "intent is detected at publish time, not configured" model means the only way to ship a version with basemaps is to ship them for the entire layer. Mixed states would let the runtime promise basemaps (via `FEATURES.PROFILE_BASEMAP`) and silently 404 for some districts — that's the failure mode the gate prevents.

Smoke-verified locally against `data/bundles/`: no PNGs → not triggered; one PNG inserted → triggers and reports 440 of 441 missing.

## Rollback posture

The feature flag is the rollback. `FEATURES.PROFILE_BASEMAP = false` returns the figure to its current tile-less state. The R2 assets remain in place but unused; the publish script can be no-op'd in the same flip. Cancelling the Stadia subscription requires deleting the cached R2 assets within a reasonable grace period to honor the "cached images valid only during active subscription" clause; the publish script's republish-from-empty mode covers the inverse (re-render everything on a fresh paid subscription).

### Rollback paths

| Failure mode | Rollback |
|---|---|
| T5 shows PNG 404s or wrong content-type | The manifest pointer hasn't flipped yet (publish-source uploads versioned artifacts BEFORE the pointer per the existing rollback design). Either re-run T4 (clean state) or restore the manifest to point at `v20260503`. |
